### Phase 1: Quick Wins (date bug + time fix)

#### GP-1: Fix incident number year bug AND make editable
**Files:** `backend/webapp/routes/reports.py`, `backend/webapp/templates/reports.html`
**Depends on:** none
**Reuses:** `_build_incident_number()` (line 115-118)
**Change:**
1. Replace `datetime.now()` with the incident date from slots:
```python
def _build_incident_number(last3, incident_date=None) -> str:
    digits = "".join(ch for ch in str(last3 or "") if ch.isdigit())[-3:]
    if not digits:
        return ""
    if incident_date:
        # Extract year-month from incident date (MM-DD-YYYY or YYYY-MM-DD)
        import re
        m = re.match(r'(\d{4})-(\d{2})', str(incident_date))
        if m:
            return f"{m.group(1)}-{m.group(2)}-{digits.zfill(3)}"
        m = re.match(r'.*-(\d{4})', str(incident_date))
        if m:
            return f"{m.group(1)}-{datetime.now():%m}-{digits.zfill(3)}"
    return f"{datetime.now():%Y-%m}-{digits.zfill(3)}"
```
2. After auto-generating, show the full number as an editable text field in the gap UI — the officer can change any part of it.
3. If the notes already contain a full incident number (e.g., "2027-07-120"), extraction should capture it and skip auto-generation.
4. Update the gap question from just "last 3 digits" to also show the generated full number for editing.
**Acceptance:** 
- Notes with 2027 date → auto-generates `2027-07-120`. 
- Officer can edit the full number before generating.
- Notes with full number already stated → uses that number, never overwrites.
**Verify:** Submit notes with 2027 date + "120" log number. Confirm generated number shows `2027-07-120` AND the field is editable. Change it to `2027-07-999`. Confirm the DOCX uses `2027-07-999`.
